## Implementation Verification Results

### Critical Issues

#### 1. `service install` command has malformed flags (`tunnel.service.ts:121`)
```typescript
// NovaPanel (BROKEN):
await run('cloudflared', ['--protocol', 'http2', 'service', 'install', tunnelToken], { sudo: true });
// Executes: sudo cloudflared --protocol http2 service install <token>
```
The `--protocol http2` flag comes AFTER the `service install` subcommand, but cloudflared's `service install` only accepts the token as an argument — not flags in that position. The command is malformed and `--protocol` is silently ignored.

**Correct alternatives:**
- `cloudflared service install <token>` (uses default auto/QUIC)
- `cloudflared tunnel run --protocol http2 --token <token>` (explicit protocol for `tunnel run`)

**Recommendation:** Either remove `--protocol http2` entirely (let cloudflared auto-select QUIC) or change to the `tunnel run` syntax if HTTP/2 fallback is needed.

---

#### 2. QUIC vs HTTP/2 default reversed (`tunnel.service.ts:121`)
The `--protocol http2` flag forces HTTP/2, but Cloudflare Tunnel's default (`auto`) prefers QUIC (HTTP/3-style) which is faster and has built-in fallback to HTTP/2. Forcing HTTP/2 removes this adaptive behavior.

**Recommendation:** Remove `--protocol http2` unless there's a specific reason to disable QUIC.

---

#### 3. Tunnel status mapping conflates "inactive" vs "down" (`tunnel.service.ts:317-326`)
```typescript
} else if (apiStatus === 'down') {
  panelStatus = 'inactive';  // "down" and "inactive" both map to 'inactive'
```
Cloudflare has four distinct statuses: `healthy`, `inactive` (never connected), `degraded`, `down` (was connected, lost contact). NovaPanel collapses `inactive` and `down` into the same `'inactive'` panel status — losing useful operational information.

**Recommendation:** Add a `down` status to `TunnelStatus` or add a `wasEverConnected` flag.

---

### Moderate Issues

#### 4. `start`/`stop` batch-update ALL tunnels in DB (`tunnel.service.ts:690-728`)
```typescript
// start() queries ALL tunnels and updates them all
const tunnels = await db.select().from(cloudflareTunnels);
for (const t of tunnels) {
  await db.update(cloudflareTunnels).set({ status: 'active' }).where(eq(cloudflareTunnels.id, t.id));
}
```
Since only ONE cloudflared systemd service runs per machine (regardless of how many `cloudflareTunnels` DB rows exist), starting/stopping the service toggles ALL tunnel DB records simultaneously. This is misleading if the intent is per-tunnel control.

**Recommendation:** Document this behavior or refactor to support multiple tunnel processes.

---

#### 5. Multiple tunnels per machine not supported (architecture-level)
The implementation assumes one cloudflared service per machine and uses hardcoded paths (`/etc/cloudflared/config.yml`, `/etc/cloudflared/<id>.json`, `/root/.cloudflared/<id>.json`). Cloudflare's docs explicitly support up to **25 replicas** of the same tunnel UUID, and the schema allows multiple `cloudflareTunnels` rows — but the systemd service model doesn't.

---

### Minor Issues

#### 6. Port scanning for metrics/config is non-deterministic if multiple instances exist
Lines 470-492 (`getTunnelConfig`) and 577-596 (`getStatus`) scan ports 20241–20250 and use the first responsive one. If multiple cloudflared instances run (supported by CF docs for replicas), the wrong instance may be queried.

#### 7. `noTlsVerify` stored as boolean, sent as `noTLSVerify` in API
`tunnel.service.ts:1078` sends `noTLSVerify: true` in the ingress `originRequest` object, which matches the Cloudflare API (per troubleshooting docs: `noTLSVerify: true`). This is correct.

## Implementation Fixes

### Fix 1: `service install` command — remove misplaced `--protocol` flag
**File:** `apps/api/src/modules/tunnel/tunnel.service.ts:121`

**Current (broken):**
```typescript
await run('cloudflared', ['--protocol', 'http2', 'service', 'install', tunnelToken], { sudo: true });
// Executes: sudo cloudflared --protocol http2 service install <token>
// --protocol appears BEFORE the subcommand and is silently ignored
```

**Fix — remove `--protocol` (let cloudflared auto-select QUIC with HTTP/2 fallback):**
```typescript
await run('cloudflared', ['service', 'install', tunnelToken], { sudo: true });
```

If HTTP/2-only is truly needed (e.g., UDP port 7844 is blocked), use `tunnel run` instead:
```typescript
await run('cloudflared', ['tunnel', 'run', '--protocol', 'http2', '--token', tunnelToken], { sudo: true });
```

**Reason:** cloudflared `service install` only accepts the tunnel token as a positional argument. Any flags must come before the `service` subcommand. Since `auto` (the default) already tries QUIC first and falls back to HTTP/2 automatically, removing `--protocol` gives better behavior with no downside.

---

### Fix 2: Tunnel status mapping — distinguish "inactive" from "down"
**File:** `apps/api/src/modules/tunnel/tunnel.service.ts:317-326`

**Current:**
```typescript
if (apiStatus === 'healthy') panelStatus = 'active';
else if (apiStatus === 'degraded') panelStatus = 'degraded';
else if (apiStatus === 'down') panelStatus = 'inactive'; // conflates "down" with "inactive"
```

**Fix — add `down` as a distinct status:**
```typescript
if (apiStatus === 'healthy') panelStatus = 'active';
else if (apiStatus === 'degraded') panelStatus = 'degraded';
else if (apiStatus === 'down') panelStatus = 'down'; // was connected, lost contact
// else 'inactive' (never connected) stays 'inactive'
```

Update the `TunnelStatus` interface and `getStatus()` accordingly.
