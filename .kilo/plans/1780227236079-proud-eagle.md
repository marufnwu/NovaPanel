# UI/Backend Logical Consistency Fix Plan — Sites, Domains & Connected Features

## Problem Statement
The NovaPanel frontend and backend are logically disconnected across site creation, site detail tabs, domain management, and domain detail tabs. Features that appear to work in the UI silently fail or return incorrect data because connected functions do not actually communicate through the API. The issues are not merely "missing routes" — entire feature flows are wired to mock data, wrong database lookups, or non-existent endpoints.

## Logical Inconsistencies Discovered

### 1. Site Creation → Domain Attachment Is Broken
**Frontend**: `CreateSiteModal` sends `primaryDomain: domainId` inside `CreateSiteInput`.  
**Backend**: `sitesService.create()` receives `data: CreateSiteInput` but **never reads `primaryDomain`**. It only inserts into `sites` table.  
**Result**: User selects a domain during creation, site is created, but domain remains unattached. User must manually attach it later.

### 2. Deployment Settings UI → Backend Does Not Exist
**Frontend**: `DeploymentSettings.tsx` is a rich editor with git repo URL, branch, build command, output dir, deploy path, auto-deploy toggles, deploy-on-PR, auto-rollback, pre/post deploy hooks, git credentials modal.  
**Hooks called**: `useDeploymentSettings`, `useUpdateDeploymentSettings`, `useDeploymentHooks`, `useTestWebhook`.  
**Backend**: No routes exist for `/sites/:id/deployment-settings`, `/sites/:id/deployment-hooks`, `/sites/:id/webhook/test`.  
**Result**: The entire "Deployments" tab configuration UI is non-functional. Save buttons call endpoints that return 404.

### 3. Environment Variables → Database Table Is Ignored
**Frontend**: `EnvVarsEditor.tsx` shows a full CRUD UI with source badges (builtin/database/.env file), secret masking, reveal toggle.  
**Backend DB**: `site_env_vars` table exists with columns `id, siteId, key, value, scope, isSystem, createdAt, updatedAt`.  
**Backend Service**: 
- `getEnvVars()` returns **hardcoded builtins** (NODE_ENV, APP_ENV, PORT) and never queries `site_env_vars`.
- `createEnvVar()` returns a mock object without inserting into the table.
- `updateEnvVar()` returns a mock object without updating the table.
- `deleteEnvVar()` only rejects builtin IDs, never touches the table.
**Result**: User can add/edit/delete env vars in the UI, but nothing persists. On refresh, only builtins appear.

### 4. Health & Stats → Data Shape Mismatches
**Frontend `SiteHealth` interface** expects:
- `uptime24h: number`, `uptime7d: number`, `uptime30d: number`
- `avgResponseTime: number`
- `errorRate4xx: number`, `errorRate5xx: number`
- `consecutiveFailures: number`
- `failures: Array<{ timestamp, error, responseCode }>`
- `healthCheckUrl: string`, `checkInterval: number`
- `lastSuccessfulCheck: string`

**Backend `getHealth()` returns**:
- `status`, `webServer`, `phpFpm`, `database`, `lastCheck`, `issues` only.
- All extended fields are missing.

**Frontend `SiteStats` interface** expects strings:
- `bandwidthToday: string`, `diskUsage: string`, `avgResponseTime: string`, `uptime: string`

**Backend `getStats()` returns** mixed types:
- `bandwidthToday: Math.floor(...) + ' MB'` (string) ✓
- `diskUsage: Math.floor(...) + ' MB'` (string) ✓
- `avgResponseTime: Math.floor(...) + ' ms'` (string) — but frontend interface says `number` for `avgResponseTime` in health
- `cpuUsage: number`, `memoryUsage: number` (not in interface)
- `uptime: "5d 3h"` (string from `formatUptime`)

**Result**: Health tab shows empty fields for uptime stats, performance, error rates, failures. Stats cards work by accident because the specific fields used happen to be strings.

### 5. SSL Management — Split Brain Between Site & Domain
**Site Detail `SettingsTab`**:
- Calls `/sites/:id/ssl` → `sitesService.getSsl()` which looks up domain by `siteId`, then cert by `domainId`.
- Returns: `{ domain, expiresAt, validFrom, autoRenew, status }` — **missing `issuer`, `daysUntilExpiry`, `sanDomains`, `domainId`**.
- Then calls SSL hooks (`useIssueLetsEncrypt`, `useRenewCertificate`, etc.) which hit `/domains/:id/letsencrypt`, `/domains/:id/renew` in `ssl.routes.ts`.
- SettingsTab expects `domainId` from SSL data to call renew/download, but `getSsl` doesn't return `domainId`.
- Workaround: SettingsTab uses `(ssl as any).domainId` which is undefined. It actually relies on `siteDomains` query to get the domain ID.

**Domain Detail `SslTab`**:
- Shows only `domain.sslStatus === 'active'` with basic "Renew" button.
- Does NOT show certificate details (issuer, valid from, days left, SAN domains, download).
- Uses `useIssueLetsEncrypt` and `useRenewCertificate` directly.

**Result**: Two different SSL UIs with different data. Site detail shows rich cert info but gets it from a simplified endpoint. Domain detail shows almost nothing. The renewal button in SettingsTab works only because it falls back to `siteDomains` query.

### 6. Database Tab → Broken Lookup Logic
**Frontend**: `DatabaseTab` calls `/sites/:id/database`.  
**Backend `getDatabase()`**:
```ts
if (!site.orgId) return null;
const [database] = await db.select().from(databases).where(eq(databases.orgId, site.orgId)).limit(1);
```
**Issue**: `sites.orgId` is optional and typically `undefined`/`null`. The lookup joins on `orgId`, not `siteId`. There is no `site_id` column in `databases` table (verify).  
**Result**: Database tab almost always shows "No database attached" even if databases exist.

### 7. Cron Jobs in SchedulerTab → Mixed Data Sources
**Frontend `SchedulerTab`**:
- Displays cron jobs from `useQuery({ queryKey: ['sites', siteId, 'cron'], queryFn: () => api.get(\`/sites/\${siteId}/cron\`) })`.
- But also uses generic `useCreateCronJob`, `useRunCronJob`, `useToggleCronJob`, `useDeleteCronJob` from `cron.ts` hooks.
- The generic `useCronJobs` hook queries `/cron?domainId=...` or `/cron`, not `/sites/:id/cron`.

**Backend**:
- `/sites/:id/cron` route exists and calls `sitesService.getCronJobs()` → `cronService.listBySite(siteId)`.
- `/cron` route exists and calls `cronService.listJobs()` (all jobs).
- There is no `/cron?siteId=` route.

**Result**: Cron jobs created via the site scheduler will have `siteId` set and appear in `/sites/:id/cron`, but the generic cron list elsewhere won't filter by site unless `domainId` is passed. The scheduler tab mixes site-scoped list with generic mutations.

### 8. Subdomain Creation → Always Targets First Domain
**Frontend `DomainsTab`**:
- `useCreateSubdomain(siteDomains?.[0]?.id || '')` hardcodes the first domain.
- User has no way to select which parent domain to create a subdomain under.
- `handleCreateSubdomain` builds full name as `${subdomainPrefix}.${parentDomainName}` where `parentDomainName = siteDomains?.[0]?.name`.

**Result**: If a site has multiple domains (e.g., `example.com` and `example.org`), subdomains are always created under the first one.

### 9. FTP Page Embedded as a Component
**Frontend `DomainDetailPage`**:
- `{activeTab === 'ftp' && <FtpPage domainId={domainId} hideDomainSelector />}`
- `FtpPage` is a **page-level component** imported from `../ftp/FtpPage`.

**Result**: A page with its own data fetching, potential navigation hooks, and side effects is rendered inside another page. This can cause routing conflicts, duplicate data fetching, and state issues.

### 10. Activity Feed → Mock Data Generation
**Frontend**: `ActivityFeed` component receives `siteId` and `siteName`.  
**Backend `getActivities()`**:
- Generates fake activities based on whether `(site as any).lastDeploymentId` exists.
- Does not query `activity_logs` table.
- Returns hardcoded strings like `"Deployment completed successfully"` and `"Site is active"`.

**Result**: Activity tab shows the same fake data for every site, regardless of actual events.

### 11. Site Detail Tab Navigation → Bypasses Router
**Frontend `handleTabChange`**:
```ts
const url = new URL(window.location.href);
url.searchParams.set('tab', tabId);
window.history.pushState({}, '', url.toString());
window.dispatchEvent(new Event('locationchange'));
```
**Issue**: Directly mutates `window.history` instead of using `navigate()` from TanStack Router. The router state may become out of sync.  
**Result**: Browser back/forward may not work correctly with tabs. Router-dependent features (active link highlighting, route guards) may break.

### 12. `SslTab` Component Is Dead Code
**Frontend `SiteDetailPage`**:
- Defines `SslTab` component (lines 765–901) with full SSL card, certificate details, auto-renew toggle, issue modal.
- `TabContent` switch statement (lines 321–352) does **not** include a case for `'ssl'`.
- The SSL UI only exists inside `SettingsTab` (lines 903–1214).

**Result**: ~140 lines of unused SSL code maintained in parallel with the active SettingsTab SSL code. Changes must be made in two places or they diverge.

### 13. HTTP Method Mismatch — Domain Make Public
**Frontend**: `useMakeDomainPublic` sends `POST` to `/domains/:id/make-public`.  
**Backend**: Route is `PUT /:domainId/make-public`.  
**Result**: 405 Method Not Allowed when user tries to make domain public.

### 14. Frontend Hook Type Duplicates
**Frontend `domains.ts`**:
- `Subdomain`, `DomainAlias`, `DomainRedirect` interfaces are defined **twice** in the same file (lines 7–21 and 38–62).

**Result**: Maintenance burden; changes may be applied to one copy and not the other.

## Remediation Plan

### Phase A — Fix Core Data Flows (Highest Impact)

#### A1. Site Creation Attach Domain
- Update `sitesService.create()` to read `data.primaryDomain` and call `attachDomain(siteId, data.primaryDomain)` after insertion.
- Update shared schema `CreateSiteInputSchema` to mark `primaryDomain` as optional but documented.

#### A2. Environment Variables Persist to DB
- Update `sitesService.getEnvVars(siteId)` to query `site_env_vars` table and union with builtins.
- Update `sitesService.createEnvVar()` to insert into `site_env_vars`.
- Update `sitesService.updateEnvVar()` to update the table.
- Update `sitesService.deleteEnvVar()` to delete from the table (reject builtins).
- Ensure returned shape matches `SiteEnvVarSchema` in shared schemas.

#### A3. Database Tab Fix
- Either:
  - Add `siteId` column to `databases` table and update `getDatabase()` to query by `siteId`, OR
  - Update `getDatabase()` to look up by `site.id` if a relation already exists.

#### A4. Health & Stats Shape Alignment
- Extend `sitesService.getHealth()` to compute and return all fields expected by `SiteHealth` interface:
  - Query deployment history for failure counts and uptime percentages.
  - Compute `avgResponseTime` from stats or defaults.
  - Populate `consecutiveFailures`, `failures[]`, `healthCheckUrl`, `checkInterval`, `lastSuccessfulCheck`.
- Fix `SiteStats` interface to match what `getStats()` actually returns, or vice versa.
  - Make `cpuUsage`, `memoryUsage`, `requestsPerMinute` part of the interface since they exist in the backend.

#### A5. Deployment Settings — Connect or Disable
- **Decision needed**: Either:
  - Add backend routes and service methods for deployment settings, OR
  - Remove/hide the deployment configuration UI until backend is ready.
- If adding: Store deployment settings in `sites` table (columns already exist: `gitRepo`, `gitBranch`, `buildCommand`, `outputDirectory`, `installCommand`, `startCommand`, `healthCheckPath`) or create a new table.
- If removing: Replace `DeploymentSettings.tsx` content with a "coming soon" card to avoid user confusion.

#### A6. Activity Feed → Query Real Logs
- Update `sitesService.getActivities()` to query `activity_logs` table if it exists, or `deployments` + `cron_history` + `auditService` logs.
- If no unified log table exists, generate activities from actual database records instead of mock data.

### Phase B — SSL Unification

#### B1. Single SSL Data Source
- Make `sslService.getCertificate(domainId)` the single source of truth.
- Update `sitesService.getSsl(siteId)` to:
  1. Find the primary domain for the site.
  2. Call `sslService.getCertificate(domainId)`.
  3. Return the full certificate shape including `domainId`, `issuer`, `daysUntilExpiry`, `sanDomains`.
- Update shared schemas to define `SiteSslInfo` shape that both frontend and backend use.

#### B2. Domain Detail SSL Tab Enhancement
- Replace the basic `SslTab` in `DomainDetailPage` with the same rich SSL view used in site detail.
- Reuse components/hooks rather than duplicating logic.

#### B3. Remove Dead `SslTab` from SiteDetailPage
- Delete the unused `SslTab` component from `SiteDetailPage.tsx`.
- Keep SSL logic only in `SettingsTab`.

### Phase C — Cron Job Consistency

#### C1. Unified Site-Scoped Cron List
- Update `SchedulerTab` to use only the site-scoped cron query (`/sites/:id/cron`) for both listing and mutations.
- OR add `GET /cron?siteId=` route to `cron.routes.ts` and update `cronService.listJobs()` to accept and use `siteId` filter.
- Ensure `useCreateCronJob` properly associates `siteId` (it already does in the hook).

#### C2. Cron Hook Alignment
- Update `useCronJobs` hook to support `siteId` filter parameter alongside `domainId`.

### Phase D — Navigation & Architecture Fixes

#### D1. TanStack Router Tab Navigation
- Replace `window.history.pushState` in `SiteDetailPage` with `navigate({ search: { tab: tabId } })`.
- Ensure `DomainDetailPage` already does this correctly (it does).

#### D2. Extract Tab Components
- Move each `SiteDetailPage` tab into `apps/web/src/components/sites/tabs/*.tsx`.
- Move each `DomainDetailPage` tab into `apps/web/src/components/domains/tabs/*.tsx`.
- Remove `FtpPage` import from domain detail; create a lightweight `DomainFtpTab` wrapper.

#### D3. Subdomain Creation Domain Selector
- In `DomainsTab`, show a dropdown to select which attached domain to create a subdomain under.
- Pass the selected `domainId` to `useCreateSubdomain` instead of hardcoding `[0]`.

### Phase E — Schema & Type Cleanup

#### E1. Shared Schema Alignment
- Update `SiteSchema`:
  - Remove `projectId` requirement; add optional `orgId`.
  - Add `'stopped'` to `status` enum.
- Update `DomainSchema`:
  - Remove `projectId` requirement; add optional `orgId`.

#### E2. Remove Duplicates
- Remove duplicate type definitions in `domains.ts` hook file.
- Remove dead `SslTab` code.

#### E3. Fix HTTP Method Mismatch
- Update `useMakeDomainPublic` to send `PUT` instead of `POST`.

## Questions / Decisions Needed

1. **Deployment settings storage**: The `sites` table already has `gitRepo`, `gitBranch`, `buildCommand`, `outputDirectory`, `installCommand`, `startCommand`, `healthCheckPath`. Should deployment settings (autoDeploy, deployOnPr, autoRollback, pre/post hooks, git credentials) be stored inline in `sites` table, or does a new `deployment_settings` table need to be created?
2. **Database-site linkage**: Should the `databases` table get a `siteId` column, or should `sites` reference a database via `databaseId`? This determines how `DatabaseTab` lookups work.
3. **Activity logs source**: Is there an `activity_logs` table that should be queried, or should `getActivities()` aggregate from `deployments`, `cron_history`, and `auditService` logs?
4. **Health check data**: Should `getHealth()` return real computed metrics (requires nginx log parsing / metrics infrastructure), or should the frontend interface be narrowed to only what the backend currently returns?

## Success Criteria
- Creating a site with a selected domain attaches that domain automatically.
- Adding an environment variable in the UI persists after page refresh.
- Deployment settings save and load without 404 errors (either by adding backend or hiding UI).
- Health tab shows all displayed fields populated (or removes fields that cannot be populated).
- SSL information is consistent between site detail and domain detail.
- Database tab shows attached database when one exists.
- Scheduler tab lists only cron jobs for the current site.
- `pnpm build` passes with zero TypeScript errors.

## Execution Order
1. Ask user for decisions on the 4 open questions.

4. Phase C (cron consistency) — independent.

7. Build & verify.
