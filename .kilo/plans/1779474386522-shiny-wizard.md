# AI Coding Agent Instruction — Modernize Single-Server Panel UI

You are modernizing an existing single-server management panel frontend.
This is NOT a rewrite. You are making targeted, incremental changes to
improve navigation, fix broken features, and make the Site the central unit.

Do NOT add multi-server, fleet management, org switching complexity,
deploy previews, or any cloud-platform features. This is a single-server panel.

Read the full codebase before making any changes.
Make one phase at a time. Do not proceed to the next phase until
the current phase is complete and working.

## CONTEXT: What this panel already has

- React SPA with Zustand for auth/org state, TanStack Query for server state
- ~30 pages covering: Sites, Databases, Domains, SSL, DNS, Mail, PHP,
  Webserver, Firewall, Cron, Backups, Monitoring, Logs, Files, Terminal,
  Containers, Registries, Webhooks, Audit, Billing, Plugins, Installer,
  Notifications, Security, Jobs, FTP, Organizations, Settings
- Sidebar with flat 25-item navigation
- JobNotificationProvider: WebSocket at /ws/jobs that invalidates TanStack
  Query caches on job completion
- Two separate alert rule systems: MonitoringPage and NotificationsPage
  (both exist, only Monitoring hits the API)
- SMTP config in NotificationsPage saves to localStorage only (no API)
- Alert Rules in NotificationsPage are client-side only (lost on refresh)
- Monitoring graphs use generateHistoricalData() — fake/synthesized data
- Backup progress modal uses fixed 1.5s intervals — not real server progress
- PHP Version save in ServerSettingsPage has a disabled Save button (TODO)
- Breadcrumb component exists but is unused on any page
- Two DataTable variants: DataTable and DataTableV2 — inconsistent usage
- ConfirmDialog requireTyping used only on FirewallPage, not elsewhere
- Audit log fetches 500 entries and filters entirely client-side

## RULES FOR ALL PHASES

- Read before writing. Understand existing component structure before
  creating new ones.
- Reuse existing components (ConfirmDialog, EmptyState, PageHeader,
  ActionDropdown, DataTableV2, modals). Do not create duplicates.
- Do not change any API endpoint paths. Only add new ones where noted.
- Do not change the auth system, Zustand stores, or QueryClient config.
- Do not change the Terminal page layout or WebSocket logic.
- After each phase: verify no existing page is broken by running through
  the critical paths (login → create site → deploy → check monitoring).
- If you find a component that already does 80% of what's needed,
  extend it rather than rewriting it.
- Leave TODO comments for anything that requires a backend change
  that isn't yet implemented, clearly marking what endpoint is needed.
- Do not silently skip a step. If something can't be done because of
  a missing API or unclear structure, stop and report what's blocking.

## Deployment Workflow

**Preferred approach (git-based):**
1. Commit all changes locally with descriptive messages
2. Push to remote repository
3. On server: `git pull` then run rebuild script
4. Server has existing `scripts/rebuild-and-deploy.sh` for this purpose

**Do NOT upload files directly via SFTP** unless explicitly requested.
Use the server's git-based workflow for all deployments.
