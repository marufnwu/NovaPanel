# Plan: Domain Verification with BIND DNS Integration

## Problem Analysis

The user reports that domain verification fails with "Could not determine domain nameservers" even when the A record is correct. The real issue is that this panel runs **BIND9** as its own DNS server, but the current verification logic only checks A records externally — it doesn't support the "panel-as-DNS-provider" model where users only need to set NS records at their registrar.

### Three Scenarios the User Asked About

| Scenario | What user means | What should happen |
|----------|---------------|-------------------|
| **A record correct, NS incorrect** | A record at registrar points to server IP, but NS is at different provider | ✅ Domain should verify and work. A record proves ownership. |
| **NS correct, no A record** | NS at registrar points to panel's nameservers (e.g., `ns1.yourpanel.com`), but no A record set yet | ✅ Domain should verify. Panel creates BIND zone and serves A record automatically. |
| **Nothing correct** | Neither A record nor NS points to panel | ❌ Domain should fail verification with clear guidance. |

### Current Code Issues

1. `verifyDomainPointsToIp()` only checks A record. It does NOT check NS delegation to panel's nameservers.
2. Domain creation (`domains.service.ts.create()`) does NOT create a BIND zone or write zone file to disk.
3. `getDomainNameservers()` relies on system resolver which may not see newly-set NS records (propagation delay).
4. No glue record verification for panel's configured nameservers (ns1/ns2 in Server Settings).

### What Other Panels Do

Some panels (like cPanel, Plesk, HestiaCP) work like this:
1. Admin configures panel nameservers (e.g., `ns1.panel.com` → server IP)
2. User sets domain NS at registrar to `ns1.panel.com, ns2.panel.com`
3. User adds domain to panel
4. Panel verifies: do domain's NS records point to my nameservers? → yes
5. Panel creates BIND zone automatically with A record
6. Domain resolves through panel's BIND

This panel should support the same workflow.

## Backend Plan

### 1. Get Panel Nameservers from Settings

**File:** `apps/api/src/modules/settings/settings.service.ts`

The panel already has `getNameserverSettings()` which returns `{ ns1, ns2 }`. We need to use these as the "panel's nameservers" for verification.

### 2. Update `verifyDomainPointsToIp()` to Support NS-Delegation Mode

**File:** `apps/api/src/utils/network.ts`

Add a new function `verifyDomainNsDelegation()`:

```typescript
export interface NsDelegationResult {
  domain: string;
  panelNs: string[];        // Panel's configured nameservers (ns1, ns2)
  domainNs: string[];         // Domain's actual NS records from registrar
  pointsToPanel: boolean;     // Does domain NS include panel's NS?
  missingNs: string[];        // Which panel NS are not in domain's NS
  error?: string;
}

export async function verifyDomainNsDelegation(
  domain: string,
  panelNs: string[]
): Promise<NsDelegationResult>
```

**Implementation:**
- Call `getDomainNameservers(domain)` to get domain's actual NS records
- Compare domain's NS with panel's configured NS (case-insensitive, strip trailing dots)
- Return whether domain's NS includes at least one of panel's NS
- If domain has no NS records → error "No NS records found"
- If domain's NS doesn't match panel's → error "Domain NS is {domainNs}, but panel NS is {panelNs}. Update NS at registrar."

### 3. Create `verifyDomainForPanel()` — Unified Verification

**File:** `apps/api/src/utils/network.ts`

This function tries BOTH verification methods and returns the best result:

```typescript
export async function verifyDomainForPanel(
  domain: string,
  serverIp: string,
  panelNs: string[]
): Promise<{
  domain: string;
  serverIp: string;
  pointsToServer: boolean;
  resolvesTo: string[];
  nsDelegation?: NsDelegationResult;
  error?: string;
  errorCode?: 'A_RECORD_WRONG' | 'NO_A_RECORD' | 'NO_NS_RECORDS' | 'NS_NOT_POINTING_TO_PANEL' | 'VERIFICATION_FAILED';
}>
```

**Logic:**
1. Try `verifyDomainPointsToIp(domain, serverIp)` first
   - If `pointsToServer: true` → return success (A record verification)
2. If A record verification fails, try `verifyDomainNsDelegation(domain, panelNs)`
   - If `pointsToPanel: true` → return success with note "NS delegation verified. Panel will create BIND zone automatically."
   - Include `nsDelegation` in result so frontend can show NS info
3. If both fail → return combined error with both A-record and NS-delegation info

### 4. Auto-Create BIND Zone on Domain Creation

**File:** `apps/api/src/modules/domains/domains.service.ts`

In `create()`, after successful verification:

```typescript
// If domain's NS points to panel, create BIND zone automatically
if (verification.nsDelegation?.pointsToPanel) {
  // Create DNS zone in SQLite
  await dnsService.ensureZone(id, data.userId, data.ipAddress);
  
  // Add A record pointing to server IP
  await dnsService.createRecord(id, {
    type: 'A',
    name: data.name,
    value: serverIp,
    ttl: 3600,
  }, data.userId, data.ipAddress);
  
  // Export zone to BIND format
  const bindZone = await dnsService.exportZone(id);
  
  // Write zone file to disk
  await bindService.writeZoneFile(data.name, bindZone);
  await bindService.addZoneToConfig(data.name);
  await bindService.reloadBind();
}
```

**Also update `delete()`** to remove BIND zone:
```typescript
await bindService.removeZoneFromConfig(domain.name);
await bindService.removeZoneFile(domain.name);
await bindService.reloadBind();
```

### 7. Fix `getDomainNameservers()` to Use Public Resolvers

**File:** `apps/api/src/utils/network.ts`

The current implementation uses system resolver which may not have domain data. Add fallback to query public DNS resolvers:

```typescript
// Method 3: Query Google DNS (8.8.8.8) directly
const googleResult = await runCmd('dig', ['@8.8.8.8', '+short', domain, 'NS'], { timeout: 10000 });
if (googleResult.stdout.trim()) return googleResult.stdout.trim().split('\n').filter(Boolean);

// Method 4: Query Cloudflare DNS (1.1.1.1) directly  
const cfResult = await runCmd('dig', ['@1.1.1.1', '+short', domain, 'NS'], { timeout: 10000 });
if (cfResult.stdout.trim()) return cfResult.stdout.trim().split('\n').filter(Boolean);
```

This bypasses local resolver issues and gets authoritative data from public resolvers.
