# Plan: Glue Record & Nameserver Verification

## Overview

Currently, the system has no verification that configured nameservers actually resolve to IP addresses (have glue records) before accepting them. This plan adds comprehensive nameserver verification at three layers:

1. **Server Settings** — Verify panel nameservers (`ns1`, `ns2`) resolve to valid IPs before saving
2. **Domain Nameservers** — Verify per-domain nameservers resolve before saving
3. **Domain Creation** — Verify the domain being added delegates to resolvable nameservers

---

## Problem Statement

### What exists now (broken / missing)

| Feature | Current Behavior | Problem |
|---------|-----------------|---------|
| Server Settings → Nameservers | Saves `ns1`/`ns2` strings to JSON file with no validation | Accepts `ns1.fake.tld` without checking if it resolves |
| Domain → Nameservers | `PUT /domains/:id/nameservers` stores array in DB with no validation | Same — no resolution check |
| Domain Creation | Checks domain A record points to server IP | Does NOT check if domain's NS records are valid/resolvable |
| Domain Detail UI | No nameserver management tab exists | Users can't view/edit domain nameservers |

### Data bug found in `domains.service.ts`

Line 512: `nameservers: JSON.stringify(nameservers)` — the schema defines `nameservers` with `{ mode: 'json' }`, so Drizzle already handles JSON serialization. Manual `JSON.stringify()` double-encodes the array into a string, corrupting the data.

## Backend Plan

### 1. New Utility: `verifyNameserverResolvable()`

**File:** `apps/api/src/utils/network.ts`

Add a function that checks whether a nameserver hostname (e.g., `ns1.example.com`) has valid A records:

```typescript
export interface NameserverVerificationResult {
  hostname: string;
  resolvesTo: string[];
  isResolvable: boolean;
  error?: string;
}

export async function verifyNameserverResolvable(hostname: string): Promise<NameserverVerificationResult>
```

**Implementation:**
- Try `node:dns.resolve4(hostname)` with 5-second timeout
- Fallback to `dig +short <hostname> A`
- Filter out private IP ranges (RFC1918, loopback, link-local)
- Return all public IPs found, or error if none
- Must reject hostnames that resolve only to private IPs

**Error messages:**
- `"Nameserver ns1.example.com does not resolve to any IP address. Add an A record for ns1.example.com at your registrar."`
- `"Nameserver ns1.example.com resolves to private IP 192.168.1.1. Glue records must point to a public IP address."`

### 2. New Utility: `getDomainDelegation()`

**File:** `apps/api/src/utils/network.ts`

```typescript
export interface DomainDelegationResult {
  domain: string;
  nameservers: string[];
  allResolvable: boolean;
  unresolvableNs: string[];
  error?: string;
}

export async function getDomainDelegation(domain: string): Promise<DomainDelegationResult>
```

**Implementation:**
- Query NS records for the domain using `node:dns.resolveNs()` + `dig +short <domain> NS` fallback
- For each nameserver, call `verifyNameserverResolvable(ns)`
- Return full delegation report

**Error messages:**
- `"Could not determine nameservers for example.com. The domain may not be registered or may not have NS records set."`
- `"Nameserver ns1.example.com for domain example.com does not resolve to an IP address."`

### 6. (Optional but recommended) Validate domain delegation on creation

**File:** `apps/api/src/modules/domains/domains.routes.ts`

In the `POST /` handler (line 20), after A-record verification (line 25–29), add an optional NS delegation check:

```typescript
const delegation = await getDomainDelegation(data.name);
if (delegation.nameservers.length === 0) {
  throw new AppError(400, 'DOMAIN_NO_NS_RECORDS', 
    `Could not find nameservers for ${data.name}. Ensure NS records are set at your registrar.`);
}
if (!delegation.allResolvable) {
  throw new AppError(400, 'DOMAIN_NS_NOT_RESOLVABLE',
    `Domain nameservers are not resolvable: ${delegation.unresolvableNs.join(', ')}`);
}
```

**NOTE:** This is strict. If the user's domain uses Cloudflare or another external DNS where the NS hostnames don't have traditional glue records (they're anycast), this might fail. The user should decide if they want this check on domain creation or only on nameserver update.

**→ QUESTION for user:** Should domain creation (POST /domains) reject domains whose NS records don't resolve, or only warn? Cloudflare NS hostnames (e.g., `lara.ns.cloudflare.com`) DO resolve, so this would pass. But some edge cases exist.

### 7. Add `GET /domains/:id/nameservers/verify` endpoint

**File:** `apps/api/src/modules/domains/domains.routes.ts`

Add a verification endpoint so the frontend can check nameservers without saving:

```typescript
fastify.get('/:domainId/nameservers/verify', async (req) => {
  const { domainId } = req.params as { domainId: string };
  const domain = await service.get(domainId);
  const nsList = domain.nameservers || [];
  const results = await Promise.all(nsList.map(ns => verifyNameserverResolvable(ns)));
  return { success: true, data: { results } };
});
```

**File:** `apps/web/src/api/hooks/domains.ts`

Add:
```typescript
export interface NameserverVerificationResult {
  hostname: string;
  resolvesTo: string[];
  isResolvable: boolean;
  error?: string;
}

export function useDomainNameservers(domainId: string) {
  return useQuery({
    queryKey: ['domains', domainId, 'nameservers'],
    queryFn: () => api.get<string[]>(`/domains/${domainId}/nameservers`),
  });
}

export function useUpdateDomainNameservers(domainId: string) {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: (nameservers: string[]) =>
      api.put(`/domains/${domainId}/nameservers`, { nameservers }),
    onSuccess: () => qc.invalidateQueries({ queryKey: ['domains', domainId, 'nameservers'] }),
  });
}

export function useVerifyDomainNameservers(domainId: string) {
  return useMutation({
    mutationFn: () => api.get<{ results: NameserverVerificationResult[] }>(`/domains/${domainId}/nameservers/verify`),
  });
}
```

## Error Messages (Central Reference)

All error messages should be user-actionable and specific.

| Code | Message | When |
|------|---------|------|
| `NAMESERVER_NOT_RESOLVABLE` | `Nameserver {hostname} does not resolve to any IP address. Add an A record for {hostname} at your registrar before using it.` | Saving panel or domain nameservers that don't have A records |
| `NAMESERVER_PRIVATE_IP` | `Nameserver {hostname} resolves to private IP {ip}. Glue records must use a public IP address.` | NS resolves to RFC1918/loopback/link-local |
| `DOMAIN_NO_NS_RECORDS` | `Could not find nameservers for {domain}. Ensure NS records are set at your registrar.` | Domain creation — no NS records found |
| `DOMAIN_NS_NOT_RESOLVABLE` | `Domain nameservers are not resolvable: {ns1}, {ns2}. Fix glue records at your registrar before adding this domain.` | Domain creation — NS hostnames don't resolve |
| `NAMESERVER_EMPTY` | `Nameserver {n} cannot be empty.` | Saving with blank nameserver field |
| `NAMESERVER_INVALID_FORMAT` | `Nameserver {hostname} is not a valid hostname. Use format: ns1.example.com` | Invalid hostname format |
