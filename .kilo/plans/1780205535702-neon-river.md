# Plan: Complete Domain Management System UI Polish & Deploy

## Goal
Finish the comprehensive domain management system by adding instruction/help text to the two primary UI surfaces (DomainDetailPage Nameservers tab and DomainsPage DNS check), then build, deploy, and manually verify the three core verification cases.

## Current State
- Backend DNS/NS verification with error codes is implemented and working.
- Nameserver glue record validation is enforced on both domain-level and server-level NS saves.
- Subdomains/aliases/redirects are persisted to DB (schema + service + routes exist and are wired).
- Frontend hooks and pages are implemented but missing user-facing instruction text.
- The edit tool has been failing on multi-line replacements; all text additions must use single-line `oldString`/`newString` edits.

## Steps

### 1. Add instruction text to DomainDetailPage.tsx Nameservers tab
**File:** `apps/web/src/pages/domains/DomainDetailPage.tsx`
- Insert a single-line help paragraph inside the Nameservers `<Card>` but before the conditional content (before line 369: `{nameserversData?.nameservers ...`).
- Text should explain what nameservers are and that glue records are verified automatically.
- Use a single-line `oldString` that includes surrounding context to keep the edit unique.

### 2. Add instruction text to DomainsPage.tsx DNS check section
**File:** `apps/web/src/pages/domains/DomainsPage.tsx`
- Insert a single-line help paragraph inside the Add Domain `<Modal>` after the Check DNS input row and before the `{dnsCheckResult && (` block.
- Text should explain that the domain A record must point to this server's IP before the domain can be added.
- Use a single-line `oldString` with surrounding context.

### 3. Build the project
Run:
```bash
pnpm build
```
This builds schemas → api → web via turbo.

### 4. Deploy to server
Run the deploy script:
```bash
scripts/rebuild-and-deploy.sh
```
Or manually copy `apps/api/dist/` and `apps/web/dist/` to the server and restart the API service.

### 5. Manual verification tests
Test each of the three cases on the deployed server:

| Case | Expected Behavior |
|------|-------------------|
| **Domain with correct A, wrong NS** | Adding the domain should succeed because A record is the authoritative routing source. |
| **Domain with correct NS, no A** | Adding should fail with error code `NO_A_RECORD` and UI message "Add an A record at your registrar pointing to the server IP." |
| **Domain with nothing set** | Adding should fail with error code `NO_NS_RECORDS` or `A_RECORD_WRONG` and UI message "Set nameservers at your registrar or wait for propagation." |

Also verify:
- Server Settings → Nameservers: verify buttons show green/red inline results.
- Domain Detail → Nameservers tab: verify button shows detailed per-NS results; edit modal validates before saving.

## Risks & Notes
- **Edit tool limitation:** Only single-line replacements should be used. If a multi-line string is needed, break it into multiple sequential single-line edits.
- **"Verify Before Saving" behavior:** The edit modal's verify button currently calls `GET /:domainId/nameservers/verify`, which validates the *currently saved* nameservers, not the ones in the textarea. This is acceptable for this plan but should be improved later to validate the edited list (e.g., by calling `POST /settings/nameservers/verify` per hostname or adding a bulk verify endpoint).
- **Rebuild required:** The user previously deployed but did not see UI changes because a rebuild was skipped after editing `ServerSettingsPage.tsx`. This plan explicitly includes rebuild + deploy to prevent the same issue.

- [ ] `DomainsPage.tsx` contains visible instruction text in the Add Domain modal.
- [ ] `pnpm build` completes with zero errors.
- [ ] Deployed UI reflects the instruction text changes.
- [ ] All three manual test cases behave as specified.
