# Plan: Fix Domain Detail Page — All Tabs, UI & Backend API Issues

## Goal
Review and fix all issues on the Domain Detail page (frontend tabs, backend API routes, and data flow) so every tab loads correctly without 404/500 errors.

## Issues Discovered

### A. Backend Route Issues

1. **Route ordering in `domains.routes.ts`**
   - `GET /:id` was registered before `GET /:domainId/subdomains`, `/:domainId/aliases`, `/:domainId/redirects`, etc.
   - Fastify matches routes in registration order; the generic `/:id` wildcard was intercepting requests meant for nested routes.
   - **Status:** Already fixed by moving specific routes before generic `/:id` routes. Needs API rebuild + restart.

2. **Missing domain-scoped Cloudflare proxy routes**
   - Frontend calls `GET /domains/:domainId/cloudflare/dns`, `/cloudflare/ssl`, `/cloudflare/firewall`, `/cloudflare/redirects`
   - These routes do **not** exist. The real Cloudflare routes are under `/cloudflare/zones/:zoneId/...`.
   - **Fix:** Add proxy routes in `domains.routes.ts` that call the Cloudflare service using the domain's zone ID.

3. **`POST /domains/:id/letsencrypt` email validation**
   - Frontend sends `email: ''` which fails Zod validation.
   - **Fix:** Frontend must collect email from user input before calling the API.

4. **`GET /domains/:id/logs/*` returns 404 when logs don't exist**
   - The `getLogs()` service throws `AppError(404, 'LOGS_NOT_FOUND')` when no log files are found.
   - This is correct for a standalone endpoint but noisy for the overview tab.
   - **Fix:** Change `getLogs()` to return `{ accessLog: '', errorLog: '' }` instead of throwing, or add a `throwOnMissing` flag.

### B. Frontend Issues

1. **Tab data is fetched eagerly on page load**
   - Every tab's `useQuery` hook runs immediately when the page mounts, even for inactive tabs.
   - This causes unnecessary API requests and error noise.
   - **Fix:** Move tab-specific data fetching into the tab render blocks, or add `enabled: activeTab === 'tabId'` to hooks.

2. **SSL "Issue Certificate" button sends empty email**
   - `issueCert.mutate({ domainId, email: '', challengeType: 'http-01' })`
   - The API schema requires a valid email.
   - **Fix:** Add an email input field to the SSL tab and pass it to the mutation.

3. **Tab navigation bypasses TanStack Router**
   - `handleTabChange` uses `window.history.pushState` + `window.dispatchEvent(new Event('locationchange'))`
   - This is a hack that doesn't integrate with TanStack Router's state.
   - **Fix:** Use TanStack Router's `useNavigate` + `search` params properly.

4. **Cloudflare tab renders firewall data table even when endpoint 404s**
   - No error boundary or empty-state handling for missing Cloudflare zone data.
   - **Fix:** Add proper loading/error states for Cloudflare sub-sections.

### C. Service/Data Issues

1. **`getCloudflareZone()` returns dummy data**
   - Returns hardcoded object with `zoneId: null`, `accountId: null`.
   - The frontend expects real Cloudflare zone info.
   - **Fix:** Link the domain to a Cloudflare zone record and return real data, or show "not linked" state.

### Step 7: Fix domainId extraction to use proper TanStack Router API
**File:** `apps/web/src/pages/domains/DomainDetailPage.tsx`

The current `useLocation()` workaround is fragile. Since `DomainDetailPage` IS the route component, `useParams()` should work. The earlier error was likely from a stale build or a different issue.

Change back to:
```typescript
const params = useParams({ strict: false });
const domainId = params.domainId;
```

Or if TanStack v1 requires `from`, use the route ID approach from the router file.

Actually, looking at `SiteDetailPage.tsx`, it uses `useLocation()` to parse the path directly. That is the established pattern in this codebase. So keep `useLocation()` but make it more robust:

```typescript
const location = useLocation();
const domainId = location.pathname.split('/').pop() || '';
```

Wait, `location.pathname` for `/domains/123?tab=overview` is `/domains/123`, so `split('/').pop()` gives `123`. But for `/domains/123/subpage` it would give `subpage`. The current `split('/domains/')[1]` is actually correct as long as the URL is `/domains/:id`.

Keep the current approach but add validation:
```typescript
const pathParts = location.pathname.split('/domains/');
const domainId = pathParts[1]?.split('/')[0] || '';
```
