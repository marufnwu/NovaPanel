# Plan: Complete UI Analysis of NovaPanel Frontend

## Objective
Perform a comprehensive, read-only analysis of the NovaPanel frontend codebase (`apps/web/src/`) to produce an exhaustive UI analysis report following the 11-section framework provided by the user. Zero code changes will be made.

## Scope

### Files to Analyze
- **Router**: `apps/web/src/router.tsx` — complete page inventory
- **Shared Components**: `apps/web/src/components/**/*.tsx` — layout, UI primitives, tables, dialogs, toasts
- **Shared Hooks/Stores**: `apps/web/src/api/hooks/*.ts`, `apps/web/src/store/*.ts`, `apps/web/src/lib/*.ts` — data fetching, auth, theme, breadcrumbs, toast
- **Pages**: `apps/web/src/pages/**/*.tsx` — every page, modal, and sub-component
- **API Client**: `apps/web/src/api/client.ts` — request/response patterns, error handling
- **Backend Routes** (for comparison): `apps/api/src/modules/**/*.routes.ts` — identify missing frontend features

### Pages Inventory (40+ routes)
Login, Dashboard, Sites, SiteDetail, Databases, DatabaseDetail, Domains, DomainDetail, SSL, DNS, PHP, Webserver, Firewall, Backups, Monitoring, Cron, Mail, Logs, Files, Terminal, Containers, ContainerDetail, Registries, Webhooks, Audit, Billing, Plugins, Installer, Notifications, Security, Jobs, FTP, Organizations, ServerSettings, Profile, ApiTokens, Services, Processes

## Methodology

### Phase 1 — Inventory & Shared Infrastructure (Section 1 & 2)
1. Read `router.tsx` to catalog all routes and their mapped components.
2. Read all shared layout components (`AppLayout`, `Sidebar`, `Topbar`, `StatusBar`, `Breadcrumb`).
3. Read all shared UI components (`DataTable`, `EmptyState`, `ErrorState`, `ConfirmDialog`, `Modal`, `ToastProvider`, `StatusBadge`, `Skeleton`, `StatCard`, `Tabs`, `Button`, `Input`, `Card`, `UploadZone`, `CommandPalette`).
4. Read shared utilities (`toast.ts`, `utils.ts`, `motion.ts`, `breadcrumb-store.ts`, `breadcrumb-context.tsx`, `theme-provider.tsx`).
5. Read `api/client.ts` and key stores (`auth.store.ts`, `theme-store.ts`, `command-palette-store.ts`).
6. Document shared infrastructure behavior, props usage, and consistency across callers.

### Phase 2 — API Hooks Deep Dive (Supports Sections 3–7)
1. Read all 37 API hook files to catalog:
   - Every `useQuery` call (query key, endpoint, response shape)
   - Every `useMutation` call (endpoint, payload shape, success/error handlers)
   - Cache invalidation patterns (`queryClient.invalidateQueries`)
   - Toast usage on mutations
   - Confirm dialog usage
2. Cross-reference hook endpoints with backend route files to identify:
   - Frontend endpoints that don't match backend routes
   - Backend routes with no frontend hook (missing features)

### Phase 3 — Page-by-Page Analysis (Section 3)
For each page, in route order:
1. Read the page component and all child components/modals it imports.
2. Trace every `useQuery` to its hook and endpoint.
3. Trace every user interaction (button, form, toggle) through:
   - Component handler → mutation hook → API endpoint
   - Success path: toast, cache invalidation, UI update, navigation
   - Error path: toast, error message, UI state
4. Document loading, empty, and error states for each data source.
5. Document missing features (backend supports but frontend lacks).
6. Document cross-page relationships (links, shared query keys).

### Phase 4 — Cross-Cutting Analysis (Sections 4–7)
1. **4A Missing States**: Tabulate loading/empty/error state coverage per page.
2. **4B Mutation Feedback**: Tabulate every mutation for loading toast, success toast, error toast, cache invalidation.
3. **4C Fake Data**: Identify hardcoded/synthesized data in any component.
4. **4D Broken Connections**: Identify features with broken API calls, wrong payloads, or incorrect response handling.
5. **4E Missing Cross-Page Relationships**: Identify features that should be reachable from related pages.
6. **4F Navigation Dead Ends**: Identify pages/features not reachable from sidebar or other navigation.
7. **4G Inconsistencies**: Identify pattern deviations (confirm word, empty states, error states, modal structures).
8. **Section 5 Feature Dependency Map**: Document prerequisites and cross-page effects for major features.
9. **Section 6 User Journeys**: Trace 6 complete user journeys end-to-end.
10. **Section 7 Component Relationship Map**: Map query key readers and writers.

### Phase 5 — Severity, Scoring & Recommendations (Sections 8–11)
1. **Section 8**: Rate every issue Critical/High/Medium/Low with user impact.
2. **Section 9**: Calculate health scores for feature completeness, data integrity, UX completeness, navigation, consistency.
3. **Section 10**: Acknowledge what works well.
4. **Section 11**: Prioritize fix order based on severity and dependencies.

## Deliverable
A single comprehensive markdown report written to a file (location TBD by user, e.g., `UI_ANALYSIS_REPORT.md`) containing all 11 sections with specific file paths, line numbers, and actionable descriptions.

## Constraints
- **Read-only**: No file edits, no test runs, no builds.
- **Evidence-based**: Every claim must reference specific code read.
- **Trace completely**: Every button click must be traced through the full chain to the API call and back.
- **Real-time documentation**: Findings will be documented as each file is read to prevent information loss.

## Estimated Effort
This analysis will require multiple sequential turns due to the large number of files (~100+ source files to read). The plan is to work methodically through the phases above.
