## Proposed Plan: Modern Server Panel Completion

### Phase 1: Frontend-Backend Parity (Close Remaining Gaps)

#### 1.1 Projects Module
**Backend:** Already has full CRUD service + routes  
**Frontend gaps:** No hooks, no page, no sidebar nav, no route

- Create `apps/web/src/api/hooks/projects.ts` (useProjects, useProject, useCreateProject, useUpdateProject, useDeleteProject)
- Create `apps/web/src/pages/projects/ProjectsPage.tsx` (list, create/edit modal, environment badge)
- Add `/projects` route to `router.tsx`
- Add "Projects" nav item to Sidebar under "Web" or "Overview" section
- Link projects to site creation flow (site form should allow selecting a project)

#### 1.2 Container Registries Module
**Backend:** Already has full CRUD service + routes  
**Frontend gaps:** No hooks, no page, no sidebar nav, no route

- Create `apps/web/src/api/hooks/registries.ts`
- Create `apps/web/src/pages/registries/RegistriesPage.tsx` (list providers: dockerhub/ghcr/ecr/gcr/selfhosted, credential management)
- Add `/registries` route to `router.tsx`
- Add "Registries" nav item to Sidebar under "System" or "Web"

#### 1.3 Background Jobs Module
**Backend:** REST routes + WebSocket exist  
**Frontend gaps:** No hooks for job listing, no dedicated page

- Create `apps/web/src/api/hooks/jobs.ts` (useJobs, useJob, useCancelJob)
- Create `apps/web/src/pages/jobs/JobsPage.tsx` (job queue list, status badges, cancel action, real-time updates via existing JobNotificationProvider)
- Add `/jobs` route to `router.tsx`
- Add "Jobs" nav item to Sidebar under "Tools"

#### 1.4 Organization Management Page
**Backend:** Full CRUD + member management  
**Frontend gaps:** Only sidebar switcher exists, no management UI

- Create `apps/web/src/pages/organizations/OrganizationsPage.tsx` (list orgs, create org, invite members, manage roles, switch org)
- Create `apps/web/src/api/hooks/organizations.ts` hooks (already exist but page missing)
- Add `/organizations` route to `router.tsx`
- Add "Organizations" nav item to Sidebar under "Account"

### Phase 2: Execution Engines (Make It Actually Work)

#### 2.1 Cron Job Execution
**Current state:** Cron expressions are stored but may not execute  
**Gap:** No cron runner/parser that actually executes commands

- Implement `apps/api/src/services/cron-runner.ts` using `node-cron` to parse cron expressions and execute stored commands
- Hook into existing cron service to read schedules from DB
- Record execution output and status back to cron job record
- Show last run / next run times in CronPage

#### 2.2 Monitoring Alert → Notification Pipeline
**Current state:** Alert rules evaluate but don't send notifications  
**Gap:** No wiring between alert evaluation and notification delivery

- In `monitoring.service.ts`, when alert rule triggers, call `notificationsService.send()` or enqueue a notification
- Create notification channels: in-app, email (via SMTP), webhook
- Add "Notification Channels" configuration to NotificationsPage

#### 2.3 WAF Rule Application to Nginx
**Current state:** WAF rules stored in DB but not applied  
**Gap:** No nginx config generation from WAF rules

- In `security.service.ts`, on rule create/update/delete, regenerate nginx WAF config (rate limiting, IP allowlists, geo blocking)
- Use `nginxService` to test and reload config
- Show "Applied" status on WAF rules in SecurityPage

#### 2.4 Plugin Loader / Executor
**Current state:** Plugins are CRUD-only  
**Gap:** No runtime loading or execution

- Implement dynamic `import()` or VM-based plugin loading from `plugins` table
- Define plugin lifecycle: `init()`, `registerHooks()`, `onConfigChange()`
- Add "Loaded" status indicator in PluginsPage
- Add hook system so plugins can extend API routes, add sidebar items, register cron jobs

#### 2.5 Backup Scheduler
**Current state:** Manual backup trigger exists  
**Gap:** No automated scheduled backups

- Extend backup service to accept schedule expressions
- Integrate with cron runner for scheduled backups
- Support retention policies (keep last N backups)
