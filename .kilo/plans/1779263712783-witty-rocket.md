# NovaPanel Architectural Modernization Plan

## Goal
Continue the backend/frontend architectural unification started in the previous session. Focus on structural improvements: shared types, migration tooling, API consistency, and eliminating type drift between frontend and backend. No CSS or design changes.

## Current State
- **Backend**: Fastify + Drizzle ORM + SQLite. Modular pattern `*.routes.ts`, `*.schema.ts`, `*.service.ts` partially adopted.
- **Frontend**: React + TanStack Router/Query + Vite. Manually duplicated TypeScript interfaces that mirror Drizzle schemas.
- **Monorepo**: pnpm workspaces with `apps/*` only. No shared packages.
- **Migrations**: Raw SQL files applied via `sqlite3` CLI because drizzle-kit fails with ESM `.js` imports.
- **Builds**: Both API and Web builds pass successfully.
- **Deployment**: Manual build + SCP to Ubuntu server (`192.168.0.211`).

## Phase 1: Shared Schema Package (`packages/schemas`)
**Priority: Critical**

Create a `packages/schemas` package that exports Zod schemas and TypeScript types derived from the Drizzle schema. Both `apps/api` and `apps/web` will import from it.

### Why
- The frontend currently manually defines `Site`, `Domain`, `RuntimeConfig`, `CreateDomainInput`, etc. in `apps/web/src/api/hooks/*.ts`. These are already stale:
  - `CreateDomainInput` still has `websiteMode` and `websiteName` (old architecture)
  - Frontend `Domain` is missing `redirectType`, `updatedAt`
  - `RuntimeConfig` on frontend lacks `docker` runtime details present in backend
- Any schema change requires manually syncing two files. This is guaranteed to drift.

### Implementation
1. Update `pnpm-workspace.yaml` to include `packages/*`.
2. Create `packages/schemas/package.json` with `type: "module"`, exports for `./sites`, `./domains`, `./auth`, etc.
3. Add Zod schemas that match the Drizzle SQLite tables:
   - `sites.ts` - `SiteSchema`, `NewSiteSchema`, `RuntimeConfigSchema`
   - `domains.ts` - `DomainSchema`, `NewDomainSchema`, `CreateDomainInputSchema`
   - `auth.ts` - `LoginSchema`, `UserSchema`
   - Shared enums as Zod native enums
4. Export inferred TypeScript types from each schema file.
5. Refactor `apps/api` to import Zod schemas from `packages/schemas` for route validation (replacing inline Zod or moving schema files).
6. Refactor `apps/web/src/api/hooks/*.ts` to import types from `packages/schemas` and delete manual interface definitions.
7. Update `apps/web/tsconfig.json` and `apps/api/tsconfig.json` path aliases to resolve `@serverforge/schemas`.

## Phase 2: Migration Runner
**Priority: High**

Replace the manual `sqlite3` CLI migration workflow with a proper Node.js migration runner.

### Why
- Manual CLI execution is error-prone and not reproducible in CI/CD or on first server startup.
- The `_journal.json` was previously hand-edited to add entries.

### Implementation
1. Create `apps/api/src/db/migrate.ts` - a standalone runner that:
   - Reads `db/migrations/meta/_journal.json`
   - Executes `.sql` files in order via the Drizzle connection
   - Tracks applied migrations in `_journal.json`
   - Handles the ESM `.js` extension issue (run inside the app context, not drizzle-kit)
2. Wire it into `apps/api/src/db/index.ts` so migrations auto-run on startup (with a `--skip-migrate` env flag for manual control).
3. Remove the need for external `sqlite3` CLI.
4. Update AGENTS.md with the new migration command.

## Phase 4: Standardize API Route Registration
**Priority: Medium**

In `apps/api/src/routes.ts`, route prefix registration is inconsistent:
- Some modules: `prefix: '/api/v1/sites'` (correct)
- Others: `prefix: '/api/v1'` and routes define full paths like `/webserver/...`, `/php/...`, `/ssl/...`

### Implementation
1. Refactor `webserver.routes.ts`, `php.routes.ts`, `ssl.routes.ts`, `dns.routes.ts`, `mail.routes.ts`, `ftp.routes.ts`, `files.routes.ts`, `cron.routes.ts`, `firewall.routes.ts`, `backup.routes.ts`, `audit.routes.ts`, `settings.routes.ts`, `notifications.routes.ts`, `installer.routes.ts`, `tokens.routes.ts`, `cloudflare.routes.ts` to use consistent module-level prefixes.
2. Update `routes.ts` so every module registers with `prefix: '/api/v1/<module-name>'`.
3. Update frontend hooks to match the new paths.

## Phase 5: Clean Up Old Architecture References
**Priority: Medium**

After unifying sites/domains, some old terminology remains in frontend code.

### Implementation
1. In `apps/web/src/api/hooks/domains.ts`:
   - Remove `websiteMode` and `websiteName` from `CreateDomainInput` (old v3 architecture)
   - Remove `Subdomain` and `DomainAlias` interfaces (tables no longer exist)
2. Search the entire `apps/web` source for "website" (case-insensitive) and replace with "site" where it refers to the old architecture.
3. Ensure `sites.ts` hook aliases (`useWebsites`, etc.) are either removed or clearly marked as deprecated.

## Phase 6: Runtime Lifecycle Completeness
**Priority: Medium**

Verify and complete the runtime manager's ability to handle all declared runtime types.

### Implementation
1. In `runtime-manager.ts`:
   - Verify `docker` runtime handling (it's in the enum but may not be implemented)
   - Ensure `updateRuntime()` properly merges partial configs
2. In `nginx.renderer.ts`:
   - Verify Node.js and Python reverse proxy blocks are correctly generated
   - Add missing `static` site optimizations
3. In `process-manager`:
   - Ensure `systemd.manager.ts` is as complete as `pm2.manager.ts`
   - Add process cleanup on site deletion

## Phase 7: Deployment Script
**Priority: Low**

Automate the build-and-deploy workflow to the Ubuntu server.

### Implementation
1. Create `scripts/deploy.ts` (or shell script) that:
   - Runs `pnpm build` for both apps
   - Uses `scp` or `rsync` to copy `apps/api/dist/` and `apps/web/dist/` to the server
   - Restarts the `novapanel.service` via SSH
2. Update AGENTS.md with the deploy command.
