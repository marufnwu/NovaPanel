# Feature-Aware Page Testing Plan

## Acknowledgment of Previous Failure

The initial attempt produced worthless smoke tests that only verify:
- Page titles render
- Empty states display
- Buttons exist

This violates the core principle of the prompt. **Smoke tests prove nothing about whether the system actually works.** They are a waste of time.

## Correct Approach

Before writing any test, we must:
1. **Read and analyze** every page component, every hook, every modal, and every backend endpoint
2. **Document** a feature map for each page that specifies the exact API call, payload, response, backend effect, and success criteria
3. **Write tests** that verify the **actual server-side outcome** of each feature
4. **Validate** that each test answers "yes" to: "If this passes, can I be confident the feature works end-to-end?"

## Test Pattern (per feature)

For each feature, the test must:

1. **Setup**: Seed required data via API (not UI)
2. **Action**: Perform the UI action (click, fill, submit)
3. **Assert UI**: Verify success feedback (toast, state change)
4. **Assert Backend**: Call the API directly to verify the server state changed
5. **Cleanup**: Remove created resources

Example for "Create Site":
- Setup: ensure a domain exists
- Action: open modal, fill form, submit
- Assert UI: toast shows success, new site appears in table
- Assert Backend: GET /sites returns the new site with correct fields
- Cleanup: DELETE /sites/:id

## What Makes a Test "Done"

Every test must answer YES to:
- [ ] Does it verify the actual server-side outcome?
- [ ] Can I be confident the feature works end-to-end?
- [ ] Does it verify correct data was sent to backend?
- [ ] Does it clean up what it creates?
- [ ] Does it work if run multiple times?
- [ ] Does it verify failure behavior?

## Execution Strategy

1. Read the full page component
2. Read every hook it uses (useQuery, useMutation)
3. Read every modal/dialog component
4. Read the backend route and service files for each endpoint
5. Write the feature map
6. Write the tests from the feature map
7. Run the tests
8. Fix failures
9. Move to next page

Do not skip any page. Do not skip any feature. Do not write a test until the feature map is complete.

## Detail Page Testing Strategy

Detail pages require additional mocking beyond list pages:

### 1. Route Parameter Mocking

The `useParams` mock must return the correct ID for the detail route:

```typescript
vi.mock('@tanstack/react-router', async () => {
  const actual = await vi.importActual('@tanstack/react-router') as any;
  return {
    ...actual,
    useParams: () => ({ siteId: 'test-site-id' }), // or databaseId, domainId
    useSearch: () => ({ tab: 'overview' }), // or other tab
  };
});
```

### 2. Tab Navigation in Tests

Tabs use `window.history.pushState` + `window.dispatchEvent(new Event('locationchange'))` to change tabs. In tests, we must simulate this by:
- Re-rendering with different `useSearch` mock values, OR
- Triggering the tab button click and relying on the event dispatch (jsdom supports `window.history.pushState`)
