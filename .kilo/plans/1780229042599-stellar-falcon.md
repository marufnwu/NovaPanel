# UI/UX & Feature Consistency Audit — Sites & Domains

## Objective
Perform a deep audit of the Sites/Websites and Domains sections of NovaPanel. Document all UI inconsistencies, feature conflicts, backend/frontend mismatches, navigation/flow issues, and code-level signals. Do not fix anything yet.

## Scope
- **Sites section**: creation flow, detail page, all tabs/sub-features
- **Domains section**: listing, creation, detail page, all tabs/sub-features
- **Sites ↔ Domains overlap**: interactions, shared responsibilities, conflicting state

## Approach
1. Map routes and components for Sites and Domains in the web app
2. Examine API routes, schemas, and services for Sites and Domains
3. Compare counterpart pages (site creation vs domain creation, detail tabs, etc.)
4. Check for UI inconsistencies, feature conflicts, backend mismatches, flow issues
5. Compile structured audit report grouped by area, with severity ratings

# AUDIT REPORT

## DATA INTEGRITY RISKS

1. **Orphaned database creation from Site detail**
   - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx` (`DatabaseTab`), `apps/api/src/modules/sites/sites.service.ts` (`getDatabase`)
   - **Issue type**: Backend/frontend mismatch + data integrity risk
   - **Description**: The `DatabaseTab` creates a database via `useCreateDatabase` (which posts to `/databases`) but never passes `siteId`. The site record's `databaseId` is never updated. After creation, the site still thinks no database is attached. Refreshing the page shows "No database attached" again, even though the database exists in the system.
   - **Severity**: Data integrity risk

2. **SSL certificate download returns `[object Object]` instead of PEM content**
   - **Location**: `apps/web/src/api/hooks/ssl.ts` (`useDownloadCert`), `apps/web/src/pages/sites/SiteDetailPage.tsx` (`SettingsTab`), `apps/web/src/pages/domains/DomainDetailPage.tsx` (`SslTab`), `apps/api/src/modules/ssl/ssl.routes.ts` (`GET /ssl/domains/:id/download/:file`)
   - **Issue type**: Backend/frontend mismatch
   - **Description**: The API returns `{ success: true, data: { pem: "..." } }`, but the frontend hook types the response as `string` (`api.get<string>`). The tab components then do `new Blob([result as string], ...)` where `result` is actually `{ pem: "..." }`. This serializes to `[object Object]`, so downloaded `.pem` files contain garbage data instead of certificates.
   - **Severity**: Data integrity risk

3. **No foreign key constraints or cascade deletes in domain-related tables**
   - **Location**: `apps/api/src/db/schema/domains.ts`, `apps/api/src/db/schema/domain-records.ts`, `apps/api/src/db/schema/sites.ts`
   - **Issue type**: Data integrity risk
   - **Description**: `domains.siteId`, `domainSubdomains.domainId`, `domainAliases.domainId`, `domainRedirects.domainId`, and `siteEnvVars.siteId` are all plain `text` columns with no `.references()` constraints and no `ON DELETE CASCADE`. Deleting a site or domain leaves orphaned records (subdomains, aliases, redirects, env vars) that reference non-existent IDs. The backend services do not manually cascade deletes either.
   - **Severity**: Data integrity risk

4. **Site `attachDomain` does not validate the domain exists**
   - **Location**: `apps/api/src/modules/sites/sites.service.ts` (`attachDomain`)
   - **Issue type**: Data integrity risk
   - **Description**: `attachDomain` validates the site exists, then runs `db.update(domains).set({ siteId }).where(eq(domains.id, domainId))` without checking if the domain exists. If an invalid `domainId` is passed, the update affects 0 rows but the method returns `{ success: true }`, misleading the frontend into thinking the attachment succeeded.
   - **Severity**: Data integrity risk

5. **Randomized/fake stats and health data returned by backend**
   - **Location**: `apps/api/src/modules/sites/sites.service.ts` (`getStats`, `getHealth`)
   - **Issue type**: Backend/frontend mismatch
   - **Description**: `getStats` returns `Math.random()` values for visitors, bandwidth, disk usage, CPU, memory, requests/min, and response time. `getHealth` returns random uptime percentages, response times, and error rates. The frontend displays these as real metrics. Users will see fabricated numbers that change on every refresh.
   - **Severity**: Data integrity risk

## SITES SECTION

### UI Inconsistencies

6. **Site creation modal uses raw `<select>` elements instead of shared `Input` or `Select` component**
   - **Location**: `apps/web/src/pages/sites/SitesPage.tsx` (`CreateSiteModal`)
   - **Issue type**: UI inconsistency
   - **Description**: Domain and Runtime selectors are native HTML `<select>` tags with manually written Tailwind classes. Every other form in the app uses the shared `Input` component or similar. The styling is similar but not guaranteed to stay in sync.
   - **Severity**: Cosmetic

7. **Site list page has no search/filter, unlike Domains list**
   - **Location**: `apps/web/src/pages/sites/SitesPage.tsx` vs `apps/web/src/pages/domains/DomainsPage.tsx`
   - **Issue type**: UI inconsistency
   - **Description**: The Domains page has a debounced search input (`max-w-[300px]`). The Sites page has no equivalent search or filtering capability, despite potentially having many sites.
   - **Severity**: Functional

8. **Site detail tab buttons use inline styles mixed with Tailwind classes**
   - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx` (tab navigation)
   - **Issue type**: UI inconsistency
   - **Description**: Active tab indicator uses both `className="... bg-foreground-primary"` and `style={{ backgroundColor: 'var(--color-text-primary)' }}` on the same element. The inline style overrides the Tailwind class, creating a mixed styling approach.
   - **Severity**: Cosmetic

9. **Site detail `OverviewTab` hardcodes "SSL Status: Not configured"**
   - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx` (`OverviewTab`)
   - **Issue type**: UI inconsistency
   - **Description**: The overview card shows a static string "Not configured" for SSL status, even when the SettingsTab correctly fetches and displays the actual SSL certificate. The Overview never reflects real SSL state.
   - **Severity**: Functional

10. **Site detail `DomainsTab` uses inline custom badge components instead of shared `StatusBadge`**
    - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx` (`DomainsTab`)
    - **Issue type**: UI inconsistency
    - **Description**: `getSslStatusBadge` and `getDnsStatusBadge` are hand-written `<span>` elements with inline Tailwind classes. The rest of the app uses the shared `StatusBadge` component. Visual appearance may diverge.
    - **Severity**: Cosmetic

11. **Site detail `SchedulerTab` uses raw HTML elements instead of shared UI components**
    - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx` (`SchedulerTab`)
    - **Issue type**: UI inconsistency
    - **Description**: The log viewer uses raw `<select>`, `<input>`, `<button>`, and `<pre>` tags with custom Tailwind classes. It does not use the shared `Button`, `Input`, or `Card` components, making the log UI visually distinct from the rest of the app.
    - **Severity**: Cosmetic

12. **Site detail quick-action buttons bypass TanStack Router for navigation**
    - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx` (SSL and Logs buttons)
    - **Issue type**: Navigation/flow issue
    - **Description**: The SSL and Logs quick-action buttons mutate `window.history.pushState` directly and dispatch a synthetic `locationchange` event. TanStack Router is unaware of these navigation changes. If the router's internal state diverges, tab switching or back-button behavior may break.
    - **Severity**: Functional

### Feature Conflicts / Missing Actions

13. **Site detail `DatabaseTab` creates orphaned databases with no site attachment**
    - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx` (`DatabaseTab`)
    - **Issue type**: Feature conflict
    - **Description**: Creating a database from the site detail page posts to `/databases` without passing `siteId`. There is no API endpoint to attach a database to a site (`/sites/:id/database` only supports GET). The created database is orphaned.
    - **Severity**: Functional

14. **Site detail page has no "Back to Sites" link or breadcrumb**
    - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx`
    - **Issue type**: Navigation/flow issue
    - **Description**: There is no explicit back button, breadcrumb, or navigation link to return to the Sites list. Users must rely on the browser back button or sidebar navigation.
    - **Severity**: Cosmetic

15. **Site detail `DomainsTab` declares unused alias-related state**
    - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx` (`DomainsTab`)
    - **Issue type**: Code-level signal
    - **Description**: `showAddAlias`, `setShowAddAlias`, `newAliasName`, and `setNewAliasName` are declared at the `DomainsTab` level but never referenced. They appear to be dead code from a partially implemented alias feature.
    - **Severity**: Cosmetic

16. **Site detail `OverviewTab` regex-parses `site.runtime` to extract PHP version, but `runtime` is just the enum string**
    - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx` (`OverviewTab`)
    - **Issue type**: Backend/frontend mismatch
    - **Description**: The code does `site.runtime.match(/php[\s-]*([\d.]+)/i)?.[1] || '8.2'`, but `site.runtime` is always a simple enum value like `'php'`. The version is stored in `site.runtimeVersion`. The regex will always fall back to `'8.2'`.
    - **Severity**: Functional

17. **Site detail `OverviewTab` conditionally renders `requestsPerMinute` based on wrong field**
    - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx` (`OverviewTab`)
    - **Issue type**: Backend/frontend mismatch
    - **Description**: `{stats?.avgResponseTime && (<div>Requests/min: {stats.requestsPerMinute}</div>)}` — this only shows requests/min when `avgResponseTime` is present, which is a logic error. Both fields come from the same `stats` object.
    - **Severity**: Cosmetic

### Backend/Frontend Mismatches

18. **Site SettingsTab expects `memoryLimit` and `maxExecutionTime` from `/sites/:id/php`, but backend only returns `version`, `configPath`, and `poolPath`**
    - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx` (`SettingsTab`), `apps/api/src/modules/sites/sites.service.ts` (`getPhp`)
    - **Issue type**: Backend/frontend mismatch
    - **Description**: The PHP Configuration card in SettingsTab displays Memory Limit and Max Execution Time, but `sitesService.getPhp()` never returns these fields. The UI always shows `'—'`.
    - **Severity**: Functional

19. **Site SettingsTab expects `forceHttps`, `gzip`, and `caching` from `/sites/:id/webserver`, but backend only returns `type`, `configPath`, `enabledPath`, and `documentRoot`**
    - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx` (`SettingsTab`), `apps/api/src/modules/sites/sites.service.ts` (`getWebserver`)
    - **Issue type**: Backend/frontend mismatch
    - **Description**: The Webserver Configuration card displays Force HTTPS, Gzip, and Caching, but the backend endpoint returns none of these. The UI always shows `'—'` or `'Disabled'`.
    - **Severity**: Functional

20. **Site SettingsTab DNS records show hardcoded `'...'` as the record value**
    - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx` (`SettingsTab`), `apps/api/src/modules/sites/sites.service.ts` (`getDns`)
    - **Issue type**: Backend/frontend mismatch
    - **Description**: `getDns` returns `{ items: siteDomains.map(d => ({ ..., type: 'A', value: '...', proxied: d.proxyEnabled })) }`. The literal string `'...'` is rendered as the DNS record value. This is placeholder data that should be dynamic.
    - **Severity**: Functional

21. **Site SettingsTab hardcodes SSL issuer as "Let's Encrypt"**
    - **Location**: `apps/api/src/modules/sites/sites.service.ts` (`getSsl`)
    - **Issue type**: Backend/frontend mismatch
    - **Description**: `getSsl` always returns `issuer: "Let's Encrypt"` regardless of the actual certificate type stored in `sslCertificates.type`. The frontend displays this as fact.
    - **Severity**: Functional

22. **`useSite` hook types response as `Site`, but backend actually returns `Site & { domains: Domain[] }`**
    - **Location**: `apps/web/src/api/hooks/sites.ts` (`useSite`), `apps/api/src/modules/sites/sites.service.ts` (`get`)
    - **Issue type**: Backend/frontend mismatch
    - **Description**: The hook declares `api.get<Site>("/sites/${id}")`, but the backend appends `domains: siteDomains` to the response. The frontend casts to `any` when accessing `(site as any)?.domains?.[0]?.name` in multiple places. The type contract is broken.
    - **Severity**: Functional

23. **Site creation form casts the entire payload `as any`**
    - **Location**: `apps/web/src/pages/sites/SitesPage.tsx` (`CreateSiteModal`)
    - **Issue type**: Code-level signal
    - **Description**: The `handleSubmit` sends the payload with `as any`, suppressing TypeScript validation. While the fields happen to match the schema now, this makes the form fragile to schema changes.
    - **Severity**: Cosmetic

24. **Site detail uses `location.pathname.split('/sites/')` to extract `siteId` instead of `useParams`**
    - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx`
    - **Issue type**: Navigation/flow issue
    - **Description**: The page manually parses the URL to get the site ID, rather than using TanStack Router's `useParams` hook. This is brittle and bypasses the router's parameter handling.
    - **Severity**: Cosmetic

### Backend/Frontend Mismatches

33. **Domain detail uses raw `URLSearchParams` for tab state while Site detail uses `useRouterState`**
    - **Location**: `apps/web/src/pages/domains/DomainDetailPage.tsx`
    - **Issue type**: UI inconsistency
    - **Description**: `DomainDetailPage` reads the initial tab via `new URLSearchParams(location.search)`, but writes tab changes via `navigate({ search: { tab: tabId } })`. `SiteDetailPage` reads the tab via `useRouterState({ select: (s) => s.location.search })`. Two different patterns for the same concept.
    - **Severity**: Cosmetic

34. **Domain detail `SslTab` does not invalidate domain query after toggling auto-renew**
    - **Location**: `apps/web/src/pages/domains/DomainDetailPage.tsx` (`SslTab`)
    - **Issue type**: Backend/frontend mismatch
    - **Description**: After toggling auto-renew, only `['ssl']` is invalidated (by the hook's global `onSuccess`). The `['domains', domainId]` query is not invalidated, so the `domain.sslAutoRenew` value in `OverviewTab` remains stale.
    - **Severity**: Functional

35. **Domain list backend `total` count ignores search filter**
    - **Location**: `apps/api/src/modules/domains/domains.service.ts` (`list`)
    - **Issue type**: Backend/frontend mismatch
    - **Description**: The `list` method applies `search`, `siteId`, and `type` conditions to the `items` query, but the `total` count uses `db.select({ total: count() }).from(domains)` with no WHERE clause. Pagination metadata is incorrect when filtering.
    - **Severity**: Functional

36. **Domain detail uses `location.pathname.split('/domains/')` to extract `domainId` instead of `useParams`**
    - **Location**: `apps/web/src/pages/domains/DomainDetailPage.tsx`
    - **Issue type**: Navigation/flow issue
    - **Description**: Same brittle URL-parsing pattern as Site detail.
    - **Severity**: Cosmetic

## SITES ↔ DOMAINS OVERLAP

### Feature Conflicts

37. **SSL is manageable from both Site detail (Settings tab) and Domain detail (SSL tab) with separate caches**
    - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx` (`SettingsTab`), `apps/web/src/pages/domains/DomainDetailPage.tsx` (`SslTab`), `apps/web/src/api/hooks/ssl.ts`, `apps/web/src/api/hooks/sites.ts`
    - **Issue type**: Feature conflict
    - **Description**: The same underlying SSL certificate (stored in `sslCertificates` table, keyed by `domainId`) can be viewed and modified from two different UIs with different React Query cache keys (`['sites', siteId, 'ssl']` vs `['ssl', domainId]`). Mutations from one view do not invalidate the other's cache. For example, renewing a certificate from the Domain detail page will not refresh the SSL card in the Site detail page.
    - **Severity**: Functional

38. **Auto-renew toggle uses different state sources in Site vs Domain detail**
    - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx` (`SettingsTab`), `apps/web/src/pages/domains/DomainDetailPage.tsx` (`SslTab`)
    - **Issue type**: Feature conflict
    - **Description**: In Site Settings, the toggle reads `ssl.autoRenew` from the `/sites/:id/ssl` response. In Domain SSL tab, it reads `domain.sslAutoRenew` from the `/domains/:id` response. These are technically the same DB column (`domains.ssl_auto_renew`), but the frontend treats them as separate data sources. If they drift (e.g., due to cache staleness), the toggles can show conflicting values.
    - **Severity**: Functional

39. **Redirects exist in both Domain detail (Redirects tab) and Cloudflare (redirect rules)**
    - **Location**: `apps/web/src/pages/domains/DomainDetailPage.tsx` (`RedirectsTab`, `CloudflareTab`)
    - **Issue type**: Feature conflict
    - **Description**: Users can create server-level redirects in the Redirects tab (stored in `domainRedirects` table) AND Cloudflare redirect rules in the Cloudflare tab. There is no UI indication of which takes precedence or how they interact. Both can exist simultaneously for the same domain.
    - **Severity**: Functional

40. **DNS records exist in Site Settings tab (read-only, fake data), Domain DNS tab (read-only), and Cloudflare DNS tab**
    - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx` (`SettingsTab`), `apps/web/src/pages/domains/DomainDetailPage.tsx` (`DnsTab`, `CloudflareTab`)
    - **Issue type**: Feature conflict
    - **Description**: Three different places show DNS data for the same domain: Site Settings (hardcoded `'...'` values), Domain DNS (actual zone records but no edit capability), and Cloudflare DNS (external records). Users have no clear mental model of which system of record is authoritative.
    - **Severity**: Functional

41. **Domain creation flows differ between Sites and Domains sections**
    - **Location**: `apps/web/src/pages/sites/SiteDetailPage.tsx` (`DomainsTab`), `apps/web/src/pages/domains/DomainsPage.tsx`
    - **Issue type**: UI inconsistency / feature conflict
    - **Description**: In DomainsPage, creating a domain shows a DNS verification check with detailed feedback (nameservers, A records, error codes). In SiteDetailPage's "Add Domain" modal, there is no DNS verification UI at all — it just asks for a name and creates. The backend enforces DNS verification on both endpoints, but the UX is completely different.
    - **Severity**: Functional

42. **Subdomain creation accepts both `siteId` and `websiteId` in schema/backend**
    - **Location**: `apps/api/src/modules/domains/domains.schema.ts` (`createSubdomainSchema`), `apps/api/src/modules/domains/domains.service.ts` (`createSubdomain`), `apps/web/src/api/hooks/domains.ts` (`useCreateSubdomain`)
    - **Issue type**: Code-level signal / feature conflict
    - **Description**: The subdomain creation schema and service accept both `siteId` and `websiteId`, treating them as aliases for the same concept (`siteId: data.siteId || data.websiteId || null`). This suggests an incomplete migration from an older naming convention. The frontend sends `websiteId` from Sites and could send `siteId` from Domains.
    - **Severity**: Functional

43. **Domain deletion invalidates only `['domains']` but not attached sites**
    - **Location**: `apps/web/src/api/hooks/domains.ts` (`useDeleteDomain`)
    - **Issue type**: Data integrity risk
    - **Description**: When a domain is deleted, the hook only invalidates the domains list query. If that domain was attached to a site, the site's domain list (`['sites', siteId, 'domains']`) remains stale, still showing the deleted domain until manual refresh.
    - **Severity**: Data integrity risk

44. **Site list shows "Runtime" column; Domain list shows "Type" and "SSL" columns — no unified view of site-domain relationship**
    - **Location**: `apps/web/src/pages/sites/SitesPage.tsx`, `apps/web/src/pages/domains/DomainsPage.tsx`
    - **Issue type**: UI inconsistency
    - **Description**: The Sites list doesn't show which domains are attached. The Domains list shows a "Site" column with a "View Site" button, but the Sites list has no reciprocal "View Domains" column. Cross-navigation is one-directional.
    - **Severity**: Cosmetic

45. **Domain detail OverviewTab doesn't link to attached site**
    - **Location**: `apps/web/src/pages/domains/DomainDetailPage.tsx` (`OverviewTab`)
    - **Issue type**: Navigation/flow issue
    - **Description**: If a domain has `siteId`, the Overview tab doesn't display the site name or provide a link to navigate to it. Users must go back to the Domains list and click "View Site" there.
    - **Severity**: Cosmetic
