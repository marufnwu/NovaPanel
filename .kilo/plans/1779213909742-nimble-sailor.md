## 1. Project Identity

**NovaPanel** (internal name: `serverforge`) is a free, open-source Linux server management panel — a self-hosted alternative to cPanel and Plesk. It manages websites, databases, email, DNS, SSL, FTP, firewalls, cron jobs, backups, and more from a single web interface.

- **Port**: 8732 (non-standard, configurable via `env.PORT` which defaults to 8443 in code but README says 8732)
- **Tenancy**: Single admin per installation (no multi-tenant/subscription model)
- **License**: MIT
- **Node requirement**: >= 20.0.0 (`.nvmrc` specifies v20)

## 4. API Architecture

### Module Pattern
Each feature module in `apps/api/src/modules/` follows a consistent structure:
- `*.routes.ts` — Fastify route definitions
- `*.schema.ts` — Zod validation schemas + inferred types
- `*.service.ts` — Business logic (class-based or functional)
- `*.middleware.ts` — Auth/validation middleware (when needed)

### Route Registration
All routes registered in `apps/api/src/routes.ts` with prefix `/api/v1/`:
- Health: `GET /api/v1/health`
- Auth: `/api/v1/auth` (login, 2FA, logout, me, password reset, sessions, API tokens)
- Stats, Domains, Sites (v4), Webserver, PHP, SSL, DNS, Mail, Databases (`/api/v1/db`), FTP, Tunnel, Files, Terminal, Logs, Jobs (WS), Cron, Firewall, Backup, Audit, Settings, Notifications, Installer, Tokens, Cloudflare

### Response Format
API auto-wraps JSON responses in `onSend` hook for `/api/` routes:
```json
{ "success": true, "data": { ... } }
{ "success": false, "error": { "code": "...", "message": "...", "field?": "..." } }
```

### Error Handling
Global error handler in `server.ts` handles:
- `AppError` (custom, with statusCode, code, message, field)
- `ZodError` (validation failures with field details)
- Fastify validation errors
- Rate limit errors (429)
- Unexpected 500s

### Authentication
Dual auth in `auth.middleware.ts`:
1. Session cookie `sf_session` → validates against `sessions` table
2. Bearer token → tries new token system (`api_tokens` table with permissions) then falls back to legacy `users.apiTokenHash`

### Security Model
- Command allowlist: `executor.ts` only allows ~60 whitelisted commands (nginx, systemctl, mysql, certbot, ufw, etc.)
- Argument sanitization: rejects control characters
- No shell interpolation: commands passed as arrays to `execa`
- Retry logic: up to 3 attempts on transient errors (EBUSY, ENOENT, etc.)
- `runTrusted()` for internal ops (logged at WARN)
- Sudo file ops: `sudo-fs.ts` wraps file operations via sudo

## 5. Database Schema

### Core Tables
| Table | Purpose |
|-------|---------|
| `users` | Single admin user (role='admin'), password hash, 2FA secrets, api tokens |
| `sessions` | Session cookies with expiry, activity tracking |
| `temp_tokens` | 2FA verification flow intermediates |
| `two_factor_backup_codes` | Argon2-hashed single-use backup codes |
| `sites` | v4 architecture: minimal site identity (id, name, systemUser, homeDir, status) |
| `websites` | Legacy website container (being phased out in favor of `sites`) |
| `domains` | Unified domain table: primary, addon, parked, subdomain, redirect, mail-only |
| `subdomains` | Legacy separate table (v3 plan migrates into `domains`) |
| `domain_aliases` | Legacy alias table (v3 plan migrates into `domains` as `parked`) |
| `domain_redirects` | Path-level redirects |
| `site_runtimes` | Runtime config (PHP, Node, Python, Static, Docker) as JSON |
| `site_processes` | Process manager config (PM2), allocated ports, start commands |
| `site_states` | Deployment state tracking |
| `background_jobs` | Job queue with status, retries, progress, dedupe keys |
| `ssl_certificates` | SSL cert metadata |
| `domain_ssl_bindings` | Many-to-many domain-cert relationships |
| `databases` | MariaDB/PostgreSQL database records |
| `email` | Mail server config |
| `dns` | DNS zone records |
| `ftp` | FTP account records |
| `cron` | Cron job records |
| `tunnels` | Cloudflare tunnel records |
| `audit` | Audit log entries |
| `stats` | Server statistics snapshots |
| `backups` | Backup job records |
| `notifications` | User notifications |
| `api_tokens` | New scoped API tokens with permissions |
| `cloudflare` | Cloudflare zone/linkage records |
| `deployments` | Deployment records |
| `site_env_vars` | Site environment variables |
| `site_health_checks` | Health check configuration |
| `activity_logs` | Activity log entries |
| `subscriptions` | Deprecated (kept for migration compatibility) |

### Key Schema Notes
- `domains.type` enum: `primary`, `addon`, `parked`, `subdomain`, `redirect`, `mail-only`
- `domains.isPrimary`: boolean, at most one per website (enforced at service layer, DB has partial unique index in v3 plan)
- `domains.parentDomainId`: self-reference for subdomains and parked domains
- `domains.suspendedConfig`: stores original nginx server block when single domain suspended
- `domains.websiteId` → `websites.id` (legacy), `domains.siteId` → `sites.id` (v4)
- `siteRuntimes.runtimeConfig`: JSON blob with schema version

## 6. Frontend Architecture

### Router
TanStack Router (code-based, NOT file-based). Defined in `apps/web/src/router.tsx`:
- Root route → Outlet
- `/login` — LoginPage
- `/reset-password` — ResetPasswordPage
- Protected route (id: 'protected') with AuthGuard → all authenticated pages
- Pages: dashboard, domains, websites, sites, site detail, website detail, webserver, php, ssl, dns, mail, databases, database detail, ftp, files, terminal, cron, firewall, logs, backups, audit, settings, api-tokens, monitoring, notifications, installer, cloudflare

### State Management
- **Auth**: Zustand store with `persist` middleware (`sf-auth` localStorage key)
  - Tracks user, isAuthenticated, pendingTwoFactor, pendingUserId, sessionHash
- **Server state**: TanStack Query (staleTime: 30s, retry: 1, refetchOnWindowFocus: false)
- **Toast notifications**: Custom ToastProvider
- **Job notifications**: JobNotificationProvider for background job status

### API Client
`apps/web/src/api/client.ts`:
- Base URL: `/api/v1`
- Methods: get, post, put, patch, delete
- Credentials: 'include' (cookies)
- Auto-handles 401: clears `sf-auth` from localStorage and redirects to `/login`
- Throws `ApiError` with code, message, status, field

### Layout
- `AuthGuard` → `AppLayout` (sidebar + header + content)
- `PageErrorBoundary` wraps detail pages (sites, websites, databases, terminal, monitoring)

## 7. Key Services

### `executor.ts` — Command Execution
- Strict allowlist of ~60 commands
- Argument sanitization (rejects `\0`, `\n`, `\r`)
- Retry: up to 3 attempts on transient errors
- Timeout: 30s default, 10MB buffer
- `runTrusted()` bypasses allowlist (logged at WARN)

### `sudo-fs.ts` — Sudo File Operations
- `atomicWrite()`: temp file + mv for atomicity
- `writeFile()`, `readFile()`, `mkdir()`, `unlink()`, `symlink()`, `chmod()`, `rename()`, `remove()`, `copy()`
- All operations run via sudo through executor

### `nginx.service.ts` — Nginx Configuration
- Website-scoped config (v3 architecture): one conf per website
- Supports: primary + parked (merged server_name), addon (own block), subdomain (own block)
- Type-aware document root resolution
- PHP-FPM socket resolution (per-website or per-domain)
- SSL certificate paths
- Single domain suspension (503 block)
- Whole website suspension (all 503 blocks)
- Config test with `nginx -T` (handles read-only /run issues)
- Atomic writes with backup/rollback on test failure

### `mariadb.service.ts` — MariaDB Management
- CLI-based via `mysql`, `mysqldump`, `mysqlcheck`, `mysqladmin`
- SQL injection prevention via `escapeSqlString()` (single-quote doubling + backslash escaping)
- Operations: create/drop database, create/drop user, grant, dump, import, query, repair, optimize, clone

### `runtime-manager.ts` — Site Runtime Management
- Supports: PHP, Node, Python, Static, Docker
- Runtime config validation
- Port allocation (30000-40000 ephemeral range)
- Default start commands per runtime
- Installation checks

### `job-queue.ts` — Background Job Processing
- SQLite-based (not Redis-based despite bullmq presence)
- Polling interval: 5 seconds
- Max concurrent: 3 jobs
- Retry: up to maxRetries (default 3)
- Dedupe keys for idempotency
- Job types: nginx reload/config, SSL provision/renew, PM2 restart/stop/delete, deployment build/rollback, DNS zone create/delete/sync, runtime install, backup create/restore, reconcile site

### `scheduler.ts` + `reconciler`
- Scheduler: cron-like task runner
- Reconciler: monitors and repairs infrastructure drift (started in main())

## 8. Domain & Website Architecture (v3/v4)

### Core Concepts
- **Website/Site** = OS-level hosting container (system user, home directory, PHP config)
- **Domain** = DNS name attached to a website
- One website can have multiple domains
- Website can exist with zero domains (no nginx config written)

### Domain Types
| Type | Docroot | Own PHP | Own Logs | SSL |
|------|---------|---------|----------|-----|
| `primary` | Yes (`httpdocs/`) | Yes (inherits or overrides) | Yes | Yes |
| `addon` | Yes (`addons/{id}/httpdocs/`) | Yes | Yes | Yes |
| `parked` | No (shares primary) | No | No | Shared with primary |
| `subdomain` | Yes (`subdomains/{id}/httpdocs/`) | Yes | Yes | Yes |
| `redirect` | No | No | No | No |
| `mail-only` | No | No | No | No |

### Directory Structure per Website
```
/var/www/sites/{websiteId}/
├── httpdocs/          # primary domain
├── addons/{id}/
│   ├── httpdocs/      # addon domain
│   └── logs/
├── subdomains/{id}/
│   ├── httpdocs/      # subdomain
│   └── logs/
├── private/
├── logs/              # primary logs
├── tmp/
├── ssl/
└── backup/
```

### Nginx Config Strategy
- One file per website: `/etc/nginx/sites-available/website-{id}.conf`
- Fully regenerated from DB on any domain change
- Primary + parked share server block
- Addon and subdomain get own blocks
- Suspended domains emit 503 block

## 10. Migration & Architecture Evolution

### Current State (v3/v4 Hybrid)
- `websites` table exists (legacy) alongside new `sites` table (v4)
- `subdomains` and `domain_aliases` tables still exist but v3 plan intends to merge into `domains`
- NginxService already implements website-scoped config generation
- SitesService uses v4 `sites` table but falls back to `websites` for migration

### v3 Implementation Plan (in `plans/implementation-plan-v3.md`)
1. Add DB columns (`homeDir`, `isPrimary`, `suspendedConfig`)
2. Migrate subdomains → `domains` with `type='subdomain'`
3. Migrate domainAliases → `domains` with `type='parked'`
4. Refactor NginxService for type-aware blocks
5. Add single domain suspension
6. Add `makePrimary` promotion logic
7. Update frontend domain UI

## 14. How the System Starts

1. `main()` in `index.ts` creates Fastify server
2. Server starts on `env.PORT`/`env.HOST`
3. Scheduler starts (cron jobs)
4. Job queue worker starts (polls SQLite every 5s)
5. Reconciler starts (infrastructure drift monitoring)
6. Graceful shutdown on SIGTERM/SIGINT stops all services

In production:
- Fastify serves static files from `../web/dist` via `@fastify/static`
- SPA fallback: unmatched non-API/non-WS routes return `index.html`
